## What Section 4.3 Says

### Search phase

1. **Step 1** — Minimize \(\sum_{i=0}^{30} H(\Delta W_i)\), keeping minimal \(H(\Delta W_{16})\) and \(H(\Delta W_{18})\) to allow a local collision in message expansion.

2. **Step 2** — With \(\Delta W\) fixed, find \((\Delta A_i, \Delta E_i)\) following Figure 2(b). Use a threshold \(t_r\):

   \[
   \sum_{i=0}^{30} H(\Delta A_i) \le t_r
   \]

   Increase \(t_r\) until the solver finds a valid solution (e.g. within 72 hours). Keep the \(\Delta A\) solution.

3. **Step 3** — With \(\Delta W\) and \(\Delta A\) fixed, minimize \(\sum_{i=0}^{30} H(\Delta E_i)\).

### Correction phase

The combined characteristic is invalid. Correction:

1. **Free** \(\Delta E_5, \Delta E_6, \Delta E_7\). **Fix** all other \(\Delta E\), and fix all \(\Delta A\) and \(\Delta W\) from the search solution.
2. Add **value-transition constraints** for \(7 \le i \le 12\).

The paper also notes use of degrees of freedom in \((\Delta A_i, \Delta E_i)_{5 \le i \le 7}\) together with the value-transition model. The explicit correction recipe frees only \(\Delta E_{5,6,7}\) and keeps \(\Delta A\) fixed.

The valid final characteristic is **Table 7**.

## `find_dc_model_31_512.py` — Search Phase

### Paper Steps 1–3 vs code

| Paper step | Implemented? | Evidence |
|------------|--------------|----------|
| Step 1 (minimize \(H(\Delta W)\)) | **No** | `ss` hardcodes W5–W18 from Table 7; no HW objective over `wd_`/`wv_` |
| Step 2 (threshold on \(H(\Delta A)\)) | **No** | `obj_value` uses `yd_` (\(\Delta E\)), not `xd_` (\(\Delta A\)); no \(t_r\) loop |
| Step 3 (minimize \(H(\Delta E)\) with fixed W, A) | **Partially** | Minimizes `yd_` over steps 5–18; W fixed via `ss`, but **A is not fixed** — solved jointly with E |

```python
def obj_value(self, obj):

    temp += "0bin000000000@%s) = 0bin%s;\n" % ("yd_" + str(i) + "_" + str(j), ...)
```

`find_dc` is **not** the paper's 3-step pipeline. It is a **single run**: fixed \(\Delta W\) + minimize \(H(\Delta E)\) on the active SAT window (steps 5–18), plus shape heuristics on E11–E14:

- Exact HW on E12, E13, E14.
- Flags on E11 | E12 (? 20 bits set).

Compared to the 28-round scripts, 31-round `find_dc` is **closer** to the paper because it minimizes \(\Delta E\) (Step 3 direction) rather than \(\Delta A\). Step 2 is still absent.

There is **no automated link** from `find_dc` output to `correct_dc` — both use manually pasted strings.

### SAT window

- `start_step = 5`, `end_step = 19` ? differential constraints built for rounds **5–18** only.
- Paper Step 3 sums over \(i = 0..30\); code only optimizes the dense middle region.

### Message expansion flag

- `find_dc` uses `op8` (all 0, Method 2 / cancellation) for message expansion.
- Same as 28-round `find_dc`; not switched in correction (unlike some SHA-256 scripts).

## Summary Verdict

**Same attack, abbreviated implementation.**

The code reproduces:

- Correct attack geometry (message words, cancellation boundaries, Table 7 final characteristic).
- A working SFS collision (`sha512_31_SFS.txt`).

It does **not** faithfully implement Section 4.3 as written:

1. No separate Step 1 / Step 2 / Step 3 pipeline.
2. No \(t_r\) threshold on \(H(\Delta A)\).
3. No automated handoff from `find_dc` ? `correct_dc`.
4. Correction deviates on **free vs fixed E7** and **value transitions at step 12**.
5. Correction HW objective disabled in `correct_dc`.

The paper is the conceptual recipe; the released code is a manual, baked-in SAT artifact that reaches the published result without exposing the full search workflow.
